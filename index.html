9.1 本章定位

AXI 相关模块在当前 SSD 控制器里分成两类：控制面和数据面。

控制面是 AXI-Lite。它负责写寄存器、触发命令、配置队列、配置 Memory/PRP window、读取 done/error/counter/status。AXI-Lite 不搬运 16KB 这种数据 payload，它只告诉 SSD 控制器“接下来要做什么”。

数据面是 AXI4。它负责外部逻辑和 FPGA 内部 memory window 之间的大块数据读写。比如 16KB 写 SSD 流程中，外侧测试状态机先通过 AXI4 把测试 pattern 写入 source buffer；随后 AXI-Lite 只触发 NVMe Write 命令。SSD 真正取数据时，是根据 PRP 地址通过 PCIe CQ 发 MemRd，FPGA 再通过 CC 返回数据。

9.15 实现注意事项

第一，AXI-Lite 的命令寄存器不是数据通路。它只负责提交命令参数，不能用来搬大块 payload。真正 payload 必须走 AXI4 memory window 和 PCIe MemRd/MemWr。

第二，REG_CMD_START 是命令快照点。写 start 之后，nvme_axil_regs 会把当时的 cmd_ctrl/nsid/addr/lba/queue_hint 打包成 pending descriptor。后续再改命令寄存器，不会影响已经 pending 的那条命令。

第三，memory/PRP base/window 要保持配套。CQ_route、AXI_TLP_bridge_rd/wr、prp_ctl 都会使用 AXI-Lite 输出的 active 配置。如果只改一部分配置，地址链路会不一致。

第四，当前 memory bridge 中 ADDRESS_TRANSLATE_ENABLE = 1。SSD 看到绝对地址，内部 RAM 使用 offset。调试 ILA 时要区分 PRP1 里的地址和 AXI RAM 端口上的地址。

第五，MAX_PAYLOAD_SIZE = 128B 决定 read completion 分片上限。256-bit 数据宽度下 128B 是 4 个 beat。返回过小会增加 TLP header 开销，返回过大则可能超过 SSD/IP 当前协商的 MPS。

第六，外部 AXI4 和内部 DMA 同时访问同一个 axi_dp_ram。当前测试流程是先写 source，再发 SSD write；SSD read 完成后再读 dest 校验，避免了同一区域同时读写造成调试复杂度上升。后续真实数据流场景需要明确 buffer ownership 或做 ping-pong buffer。

第七，ssd_rw_test_ctrl 的四个 start 信号不能同拍多路触发。源码中用 start_edge_count 检查，如果一次检测到多个 start edge，会进入 ERR_MULTI_START。这让 VIO 控制更清楚：一次只启动一个阶段。

9.16 章节总结

AXI/Memory 模块把 SSD 控制器的“命令控制”和“数据搬运”拆成两条清晰路径。AXI-Lite 负责寄存器配置、命令提交和状态回读；AXI4 负责真实数据缓存；PCIe memory bridge 负责把 SSD 对 PRP 地址的 MemRd/MemWr 转换成 AXI RAM 读写。

当前实现中，axi_lite_ctl/nvme_axil_regs/cmd_source_mux 构成控制面，pcie_memory_clt_top/AXI_TLP_bridge/axi_dma/axi_dp_ram 构成数据面。16KB 测试流程通过 ssd_rw_test_ctrl 串起 AXI4 写数据、AXI-Lite 发命令、SSD 访问 memory window、AXI4 读回校验这几个步骤。对当前板级读写闭环来说，这套结构已经能够表达标准 NVMe Host memory 语义：SSD 并不直接知道 FPGA 内部 RAM，只看到一段可以按 PRP 地址访问的 host memory window。

8.15 实现注意事项

第一，PRP List window 和 memory window 要分开。当前 memory 数据窗口在 0x3000_xxxx，PRP List 虚拟窗口在 0x9000_0000，两者不会冲突。如果后续通过 AXI-Lite 改配置，也要保持二者不重叠。

第二，cfg_prp_page_size 应该使用 NVMe CC.MPS 对应的 page size。当前默认 4096，对大多数 SSD 的 4KB MPS 配置是匹配的。如果后续初始化阶段改变 CC.MPS，AXI-Lite 的 REG_PRP_PAGE_SIZE_SHADOW 也要同步更新。

第三，当前 PRP List 是连续 buffer 模型。它适合 FPGA 内部 AXI RAM 连续地址、顺序读写和固定测试窗口。如果未来要支持外部 host 侧任意离散物理页，就需要把 prp_list_ctx 从公式生成升级成真正的 PRP entry RAM 或 descriptor RAM。

第四，CID context 依赖 CID 生命周期。只要 command_record 保证未完成命令的 CID 不被重复分配，PRP context 覆盖就是安全的。当前 context 不主动清零，是因为新命令复用 CID 时会覆盖旧内容；completion 后 CID 回收由 command 模块负责。

第五，PRP completion 的 requester ID、tag、lower address 必须来自 SSD 原始 MemRd TLP。prp_cq_tlp 解析这些字段，prp_cc_tlp 原样放回 CplD header。这部分如果错位，SSD 会收不到匹配 completion，表现为写命令或读命令卡住。

第六，PRP_ENTRY_INDEX_WIDTH = 13 对应每个 CID 8192 个 entry，也就是 64KB PRP List window。这个位宽要和 CID << 16 的地址规划一起看，不能单独修改。如果要扩大单 CID PRP List window，需要同时调整地址切片和总 PRP window size。

8.16 章节总结

PRP 模块在当前 SSD 控制器里承担的是“把 FPGA 内部连续数据 buffer 转换成 NVMe 可识别地址描述”的职责。命令提交前，prp_core_calc 申请 CID 并计算 PRP1/PRP2；跨多页时，prp_list_ctx 保存 CID 到 base page/extra pages 的上下文。SSD 执行命令后，如果需要读取 PRP List，CQ_route 会把对应 MemRd 转给 PRP 模块，prp_cc_tlp 按 context 动态返回 PRP entry。

这一版 PRP 设计的重点不是做完整操作系统级页表，而是服务当前 FPGA 内部连续 buffer 的高速读写链路。它支持 4KB page 默认配置、NLB zero-based 字节数换算、跨页 PRP1/PRP2 生成、按 CID 的虚拟 PRP List window，以及 SSD 对 PRP List 的 PCIe MemRd 响应。对当前 16KB 顺序读写闭环和后续更大连续 buffer 测试，这套结构已经覆盖了关键路径。

7.15 当前实现需要注意的细节

7.15.1 队列动态增减是 enable/disable，不是运行时 Create/Delete

当前硬件里 16 路 IOSQ lane 固定存在。AXI-Lite 的 io_queue_enable 控制的是是否接收新命令，以及 lane 是否在 draining 结束后复位。真正 NVMe 协议层面的 Create/Delete IO Queue 命令仍然属于初始化/管理流程，不是在每次 enable/disable 时动态发送。

7.15.2 IOCQ 是共享队列

当前结构是 16 路 IOSQ + 1 个共享 IOCQ。IOCQ 通过 CQE 中的 SQID 把 completion route 回对应 command record port。这比每路 IO queue 都做独立 CQ 更省地址和逻辑资源，但要求 queue_cpl_route 和 queue_sq_head_demux 对 SQID 处理正确。

7.15.3 SQ head 依赖 CQE 回写

SQ lane 的 head_ptr_reg 不是自己随命令写入推进，而是来自 completion 中的 SQ head：

如果 CQE 没回来或 SQ head 没有正确 demux，SQ 会逐步变满，nvme_command_ready 会被拉低。

7.15.4 phase 必须是寄存器状态

CQ phase 不是一个单拍组合判断。当前 CQ_lane_ctl.v 使用 write_expected_phase_reg 和 expected_phase_reg 保存队列 wrap 状态，tail/head wrap 时分别翻转。这样才能正确区分同一个 CQ 地址上的新旧 CQE。

7.15.5 doorbell 聚合保存最新 ptr

queue_doorbell_coalesce 聚合的是最新 tail/head，而不是保存每一次更新。对 doorbell 来说这是合理的，但调试时会看到 doorbell 写次数少于 SQE/CQE 更新次数，这是预期行为。

7.15.6 queue_weight 为 0 当前不是禁用

queue_cmd_dispatch 和 queue_tlp_rr_arb 中，weight 为 0 会被解释为 DEFAULT_WEIGHT。如果要禁用队列，应该清 io_queue_enable，不要靠 weight=0。

7.15.7 有效地址以 SSD_ctl 参数为准

nvme_queue_ctl.v 内部有默认地址参数，但当前系统中 SSD_ctl.v 会覆盖它们。查看上板地址冲突时，要优先看 SSD_ctl.v 的 ADMIN_SQ_ADDRESS、ADMIN_CQ_ADDRESS、IO_SQ_ADDRESS、IO_CQ_ADDRESS 和 RANGE_SIZE。

7.16 章节总结

Queue 模块当前已经形成完整的 NVMe 队列层：

  • 支持 1 个 ASQ、1 个 ACQ、16 个 IOSQ 和 1 个共享 IOCQ。
  • ASQ/IOSQ 保存 512-bit SQE，并响应 SSD 对 SQ window 的 MemRd。
  • ACQ/IOCQ 接收 SSD 写回的 128-bit CQE，并解析 CID、SQID、SQ head、status、phase。
  • IO command 支持 Fixed、RR 和 Weighted RR 分发。
  • IO queue 支持 enable、live、draining 和 inflight 计数。
  • 16 路 IOSQ 地址按 base+stride route，避免多个 lane 暴露成顶层大量接口。
  • 多路 IOSQ TLP 输出通过 RR arbiter 汇聚，packet 内不交错。
  • SQ/CQ doorbell 支持 count/time 双阈值聚合。
  • completion 通过 SQID route 回 command_ctl，继续完成 CID 匹配和资源回收。

从系统角度看，Queue 模块把 NVMe 的“队列在主机内存中，SSD 主动读取 SQE/写回 CQE”这一机制硬件化。Command 模块只需要提交 SQE 并等待 completion，PRP/Memory 模块只需要响应数据访问，Queue 模块负责把中间的队列存储、doorbell、phase、head/tail 和多队列调度处理清楚。

PCIe 枚举本质上是 Root Complex 对下游设备配置空间的扫描和配置过程。FPGA 在这个项目里扮演的是主机侧逻辑，所以它需要主动发起 Configuration Read / Configuration Write，通过配置空间判断链路后面挂着什么设备、这个设备在哪个 BDF 上、BAR 空间需要多大，以及后续 NVMe 初始化应该访问哪个 MMIO 基地址。

Offset 内容 用途
0x000 Vendor ID / Device ID 判断 function 是否存在，Vendor ID = 16'hffff 表示不存在
0x004 Command Register 最后使能 Memory Space、Bus Master

5A.2 为什么要改成深度优先枚举

早期枚举模块 pcie_us_enum_state.v 更像是面向单个固定 SSD 的初始化脚本。它默认目标 BDF 接近 01:00.0，直接读 Vendor ID、探测 BAR、写 BAR 地址、写 Command Register，然后读 NVMe CAP 验证链路。

这种方式适合第一阶段上板验证，因为最短路径清楚，能快速证明 FPGA 可以通过 RQ/RC 和 SSD 配置空间通信。但它的问题也明显：

  • 如果 SSD 不在固定 BDF 上，枚举会失败。
  • 如果链路后面挂了 PCIe Switch，旧流程不会进入 secondary bus。
  • 如果挂了多个 endpoint，旧流程无法判断哪个设备才是 NVMe。
  • 从枚举完整性看，它更像固定设备 Bring-up，不像完整的 Root Complex 枚举流程。

pcie_us_enum_dfs_state.v 的目标，是把这部分升级为更标准的硬件枚举流程：从一个起始 bus 开始扫描 BDF，遇到 PCI bridge 就配置桥并优先下探它的 secondary bus，直到找到 class code 为 NVMe 的 endpoint。

4.1 四通道一句话区分

4.11 ready/valid 和流控关系

当前工程主要依赖 AXI-Stream ready/valid 背压保证 TLP 传输正确性。PCIe IP 也暴露了 credit 观测接口，例如 cfg_fc_ph/pd/nph/npd/cplh/cpld，但当前没有把它们做成完整的 credit-aware 调度。

当前实现中最重要的规则是：

状态机里的地址递增、payload beat 计数、header 切换、sop/eop 推进，都应该跟随 fire。如果 ready 被 PCIe IP 拉低，而状态机仍然继续更新 header 或 data，就会出现几类典型问题：

  • 最后一拍数据重复或丢失。
  • header 对应的是上一笔请求，data 对应的是下一笔请求。
  • eop 提前或滞后。
  • CplD length 和实际 payload beat 数不一致。
  • SSD 后续不再继续读 PRP 或 data，最终命令超时。

cfg_fc_* credit 信号可以作为后续优化方向，用来判断 PCIe IP 内部 header/data credit 是否紧张。但当前工程的基础正确性仍然优先依赖 ready/valid 和 FIFO/arbiter 的背压处理。

3.7 FPGA 与 SSD 的运行时交互流程

从 NVMe 的角度看，一条命令不是 FPGA 单向把数据推给 SSD，而是 FPGA 提交命令后，SSD 再根据队列和 PRP 地址主动访问 FPGA。

3.7.1 命令提交

FPGA 构造 64B NVMe SQE，并写入 ASQ 或 IOSQ。随后 FPGA 通过 RQ 发送 Memory Write TLP，写 SSD BAR0 中对应队列的 SQ Tail Doorbell。

这一阶段的关键是：doorbell 不能早于 SQE 可读。否则 SSD 看到 doorbell 后来读 SQE，可能读到旧内容或无效内容。

3.7.2 SSD 获取命令

SSD 收到 doorbell 后，会通过 CQ 发起 Memory Read，请求读取 FPGA 地址空间中的 SQE。

FPGA 的 CQ_route 根据地址把请求路由到 queue 模块，queue 模块取出 SQE 后通过 CC_round_robin 返回 CplD。到这里 SSD 才真正拿到 NVMe 命令。

这些地址必须在 NVMe 初始化、CQ route、queue、PRP、memory controller 中保持一致。只改一个模块会造成 SSD 请求打到错误窗口，表现为命令卡住、CQE 不回来或数据校验失败。

2.1 NVMe over PCIe 基础

2.1.1 PCIe 和 NVMe 的关系

NVMe SSD 底层通过 PCIe 传输数据，但 NVMe 和 PCIe 不是同一层协议。

在普通 PC 里，CPU 和操作系统驱动完成这些工作：

  • PCIe 枚举，找到 SSD 设备和 BAR 空间。
  • 配置 NVMe 控制器寄存器。
  • 建立 Admin Queue 和 IO Queue。
  • 把 NVMe command 写入 Submission Queue。
  • 写 doorbell 通知 SSD 取命令。
  • 解析 Completion Queue Entry，回收命令资源。

本项目把这些流程下沉到 FPGA 内部，用 RTL 完成主机侧控制逻辑。这里的边界要说清楚：FPGA 实现的是一套面向固定场景的轻量级 NVMe Host Controller，而不是完整商用 NVMe 驱动。

2.2.1 Xilinx PCIe IP 手册学习

这里需要分清三个层次：

  • sys_clk / sys_clk_gt 是 PCIe IP 的参考时钟输入，来自板卡 PCIe 专用参考时钟。

2.3.13 RQ 接口：FPGA 主动发起请求

RQ 是 Requester Request。用户逻辑通过 RQ 把请求 TLP 送进 PCIe IP。

RQ 数据包以 128-bit descriptor 开始，如果是 Memory Write 或 Message 这类带 payload 的 TLP，descriptor 后面继续跟 payload。当前 256-bit 接口下，descriptor 通常可以在第一个 beat 中放下。

当前项目中 RQ 主要承担：

  • PCIe 枚举阶段的 Config Read / Config Write。
  • NVMe 初始化阶段的 SSD BAR0 寄存器读写。
  • 写 ASQ/IOSQ/ACQ/IOCQ doorbell。
  • 某些 outbound Memory Write / Memory Read 测试路径。

实现时需要注意：

  • RQ 是 s_axis_rq_*，方向是用户逻辑到 PCIe IP。
  • s_axis_rq_tvalid && s_axis_rq_tready 成立时，当前 beat 才真正发送。
  • tready=0 时，descriptor、payload、tkeep、tlast、tuser 都要保持稳定。
  • Memory Read 通常没有 payload，但需要正确填写 tag、address、length、first_be/last_be。
  • Memory Write 需要按实际 MPS 切分 payload，当前上板测试里 MPS 按 128B 处理时，一包 128B 对应 4 拍 256-bit 数据。

在本项目中，枚举和初始化能不能继续，首先看 RQ 发出去的 Config Read / Memory Read 是否能在 RC 口收到 completion。IO 阶段能不能启动，主要看 doorbell Memory Write 是否正确从 RQ 发出。

2.3.14 RC 接口：FPGA 请求的返回 completion

RC 是 Requester Completion。凡是 FPGA 通过 RQ 发出的 non-posted request，例如 Config Read、Memory Read，对端返回的 completion 会从 RC 口进入用户逻辑。

RC 数据包以 96-bit descriptor 开始，后面可能带 completion data。无数据 completion 和带数据 completion 都走同一个接口。

当前项目中 RC 主要用于：

  • 枚举阶段读取 SSD Vendor ID、Device ID、Class Code、BAR mask 等配置空间内容。
  • NVMe 初始化阶段读取 CAP、VS、CSTS 等寄存器。
  • 检查 RQ 发出的 read request 是否有 completion 返回。

实现时需要注意：

  • RC 是 m_axis_rc_*，方向是 PCIe IP 到用户逻辑。
  • completion descriptor 中的 tag / requester id / byte count / lower address 等字段要和发出的请求对应。
  • 配置读和寄存器读返回的数据需要按 DW 顺序解析，之前上板调试中大小端和 DW 排列是容易出错的地方。
  • 如果一笔读取被拆成多个 completion，用户逻辑需要能按照 tag 和 byte count 继续接收。
  • m_axis_rc_tready 可以拉低形成 backpressure，但拉低后要保证上游状态机不会错误消费数据。

在当前工程里，RC 解析正确后，枚举状态机才能知道 SSD 是否存在、BAR0 大小是多少、CSTS.RDY 是否已经拉高。

2.3.15 CQ 接口：SSD 主动访问 FPGA 地址空间

CQ 是 Completer Request。它表示对端 SSD 主动访问 FPGA 暴露出来的地址空间。NVMe 运行时大量关键动作都从 CQ 进入 FPGA。

CQ 数据包以 128-bit descriptor 开始，Memory Write 会带 payload，Memory Read 不带 payload 但需要 FPGA 后续通过 CC 返回 completion。

当前项目中 CQ 主要承载：

  • SSD 读取 ASQ/IOSQ 中的 SQE。
  • SSD 读取 PRP List Window。
  • SSD 读取 FPGA memory window 中待写入 SSD 的数据。
  • SSD 通过 Memory Write 写回 ACQ/IOCQ completion。
  • SSD 通过 Memory Write 把读出的数据写回 FPGA memory window。

因此 CQ route 是整个工程中很关键的一层。它需要根据 TLP address 把请求分发到不同模块：

CQ 地址范围 路由目标 处理方式
ASQ / IOSQ queue 模块 SSD MemRd SQE，FPGA 通过 CC 返回 SQE 内容
ACQ / IOCQ queue 模块 SSD MemWr CQE，FPGA 更新 CQ lane 并触发 completion 处理
PRP List Window PRP 模块 SSD MemRd PRP list，FPGA 通过 CC 返回 PRP entry
Memory Window memory 模块 SSD MemRd 时返回 data，SSD MemWr 时写入 data buffer

实现时需要注意：

  • CQ 是 m_axis_cq_*，方向是 PCIe IP 到用户逻辑。
  • Memory Write 的有效字节要结合 tkeep、byte_en 或 first/last byte enable 处理。
  • Memory Read 请求不能只被接收，还要产生对应 CC completion。
  • CQ route 的地址窗口要和初始化写给 SSD 的 ASQ/ACQ/IOSQ/IOCQ/PRP/memory 地址一致。
  • 如果 CQE 写回的 SQID / CID 和 command_record 不匹配，要进入 mismatch/drop 统计，而不是直接当作正常 completion。

从调试角度看，FPGA 写 doorbell 之后，如果 CQ 口完全没有 SSD 读 SQE 的请求，通常说明 doorbell 地址、队列地址、队列 enable 或 NVMe 初始化状态有问题。

2.3.16 CC 接口：FPGA 响应 SSD 的读取请求

CC 是 Completer Completion。当 SSD 通过 CQ 发起 Memory Read，FPGA 需要通过 CC 返回 completion 或 completion data。

CC 数据包以 96-bit descriptor 开始，带数据的 completion 后面跟 payload。对于大于 MPS 的数据，需要拆成多个 completion。

当前项目中 CC 主要返回：

  • ASQ / IOSQ 中的 SQE。
  • PRP list entry。
  • memory window 中的数据。

实现时需要注意：

  • CC 是 s_axis_cc_*，方向是用户逻辑到 PCIe IP。
  • 每个 CQ Memory Read 都要有对应 completion。
  • completion status 正常时填 Successful Completion。
  • 数据长度超过 MPS 时要拆成多个 CplD。
  • s_axis_cc_tvalid && s_axis_cc_tready 成立时才推进 payload beat 计数。
  • tready=0 时 header/data/tkeep/tlast 必须保持稳定。

之前上板调试中，CC 路径最容易出现的问题是最后一拍 header/data 错位，或者 ready 拉低时状态机仍然推进，导致 SSD 收到的 CplD 长度或内容不对。这个问题通常要通过 ILA 同时抓 CQ 请求 header、CC 返回 header、payload beat count 和 ready/valid 来定位。

2.4 设计注意事项

这一节对应原来的 2D.设计注意事项.md，记录当前工程里容易在上板时暴露的问题。

2.4.1 时钟和复位边界

sys_clk / sys_clk_gt 来自板卡 PCIe 参考时钟路径；user_clk 是 PCIe IP 输出给用户逻辑的 AXI-Stream 时钟。RQ/RC/CQ/CC、NVMe 初始化、command、queue、PRP、memory 等主逻辑默认都放在 user_clk 域。用户逻辑复位跟随 user_reset，内部可转换成 user_rst_n。

如果外部 AXI 或 AXI-Lite 不在 user_clk 域，需要明确做 CDC。调试时不要只看外部复位是否释放，也要看 PCIe IP 输出的 user_reset 和 user_lnk_up。

2.4.2 Root Port 和 Endpoint 角色

当前 FPGA 是 Root Port，SSD 是 Endpoint。Xilinx IP 配置页里的 PF0 BAR 是 FPGA 这个 PCIe 端口自己的 BAR；NVMe 初始化里真正要访问的是 SSD 配置空间里的 BAR0。SSD BAR0 分配完成后，FPGA 才能通过 SSD_BAR0_BASE + offset 访问 CAP、CC、CSTS、AQA、ASQ、ACQ、doorbell 等寄存器。

2.4.3 MPS / MRRS 和数据分包

PCIe IP 配置页里填写的是能力上限，实际运行时还要看链路协商和配置后的 MPS/MRRS。上板时如果实际 MPS 是 128B，completion data 和 Memory Write payload 就要按这个边界切分。

当前用户侧 AXI-Stream 是 256-bit，一拍 32B。为了效率，数据 payload 不应长期只发 32B 小包；但每个 TLP 也不能超过实际 MPS。更合理的做法是按 MPS 聚合，例如 128B 对应 4 拍 256-bit 数据。

2.4.7 ILA 调试优先级

上板调试时建议按层级抓信号：

层级 观察点
PCIe 链路 user_lnk_up、user_rst_n、ltssm_state
初始化 枚举状态机、NVMe init 状态、queue init done
RQ/RC 配置读写、寄存器读写、doorbell、completion
CQ/CC SSD 读 SQE / PRP / data，FPGA 返回 CplD
command/queue/PRP CID、SQID、opcode、PRP 地址、completion 匹配

不要一开始只抓一个很宽的拼接总线。更适合的做法是把关键路径拆成几组 ILA，每组信号都打一拍，触发条件也围绕具体事件设置。

2.4.8 章节总结

当前版本更适合作为轻量级 NVMe Host Controller 原型：

  • 支持固定地址窗口下的 Admin Queue、IO Queue 和 16KB 读写闭环。
  • 支持基础多队列结构、CID 匹配、PRP list 响应和状态统计。
  • 不把 MSI-X、完整 PRP RAM、完整错误恢复、复杂电源管理作为当前主线。

本章可以归纳为：项目的重点是从 PCIe TLP 到 NVMe 队列的数据闭环，而不是把所有商用 SSD 驱动特性一次性全部覆盖。

1.1 项目目标

这个项目的目标是在 FPGA 内部实现一套轻量级 NVMe over PCIe Host Controller。它不是简单调用 PCIe IP，也不是只做一条 DMA 搬运通路，而是在 Xilinx PCIe Gen3 IP 提供的 AXI-Stream TLP 接口之上，用 RTL 完成 SSD 访问所需的主机侧控制逻辑。

我这里主要做的事情是：把 Xilinx PCIe Gen3 IP 输出的 AXI-Stream 数据流解析成 TLP header 和 payload，再按照地址、TLP 类型和方向路由到 NVMe 初始化、队列、PRP、数据缓存等模块。不同模块处理完成后，再通过 RQ/CC 等通路返回对应的 TLP，或者更新 FPGA 内部的 memory window，从而完成 FPGA 与 SSD 之间的数据交互。

1.2 项目背景

在普通 PC 系统中，NVMe SSD 由 CPU、操作系统和驱动共同管理。PCIe 枚举、NVMe 初始化、队列建立、PRP 地址管理、命令提交和 completion 回收都由软件栈完成。

对于数据采集、实时缓存和硬件加速这类场景，如果每次 IO 都依赖 CPU 参与，会带来软件调度开销，也会增加系统时延的不确定性。这个项目尝试把 SSD 控制链路的一部分下沉到 FPGA 内部，让 FPGA 作为 PCIe Host/Root 侧逻辑直接控制 NVMe SSD。

项目里 FPGA 负责：

  • 建立 PCIe 链路，并完成 SSD 枚举。
  • 配置 NVMe 控制器寄存器。
  • 建立 Admin Queue 和 IO Queue。
  • 维护 command、queue、PRP 和 memory window。
  • 通过 AXI-Lite 暴露控制寄存器，通过 AXI4 暴露数据缓存窗口。

这样做的直接意义是：外部采集或计算模块可以先把数据写入 FPGA 侧 AXI4 memory window，再由 FPGA 触发 NVMe Write 写入 SSD；读数据时也可以由 FPGA 触发 NVMe Read，把 SSD 数据读回 AXI4 memory window。对于图像数据、雷达原始数据等连续大带宽数据流，这种结构可以减少 CPU 参与，为后续长时间记录和回放打基础。

当前工程中几个调试接口的定位如下：

  • AXI-Lite：配置寄存器、选择命令来源、触发读写命令、读取状态和错误信息。
  • AXI4：访问 FPGA 侧数据缓存窗口，测试时由 ssd_rw_test_ctrl 写入或读出 16KB 数据。
  • VIO：板级调试阶段手动控制测试开始，例如 rw_write_start、rw_write_start_command、rw_read_start_command、rw_read_start。
  • ILA：观察 PCIe TLP、PRP、CQ route、completion、done pulse 等关键路径。
