9.1 本章定位

AXI 相关模块在当前 SSD 控制器里分成两类：控制面和数据面。

控制面是 AXI-Lite。它负责写寄存器、触发命令、配置队列、配置 Memory/PRP window、读取 done/error/counter/status。AXI-Lite 不搬运 16KB 这种数据 payload，它只告诉 SSD 控制器“接下来要做什么”。

数据面是 AXI4。它负责外部逻辑和 FPGA 内部 memory window 之间的大块数据读写。比如 16KB 写 SSD 流程中，外侧测试状态机先通过 AXI4 把测试 pattern 写入 source buffer；随后 AXI-Lite 只触发 NVMe Write 命令。SSD 真正取数据时，是根据 PRP 地址通过 PCIe CQ 发 MemRd，FPGA 再通过 CC 返回数据。

9.15 实现注意事项

第一，AXI-Lite 的命令寄存器不是数据通路。它只负责提交命令参数，不能用来搬大块 payload。真正 payload 必须走 AXI4 memory window 和 PCIe MemRd/MemWr。

第二，REG_CMD_START 是命令快照点。写 start 之后，nvme_axil_regs 会把当时的 cmd_ctrl/nsid/addr/lba/queue_hint 打包成 pending descriptor。后续再改命令寄存器，不会影响已经 pending 的那条命令。

第三，memory/PRP base/window 要保持配套。CQ_route、AXI_TLP_bridge_rd/wr、prp_ctl 都会使用 AXI-Lite 输出的 active 配置。如果只改一部分配置，地址链路会不一致。

第四，当前 memory bridge 中 ADDRESS_TRANSLATE_ENABLE = 1。SSD 看到绝对地址，内部 RAM 使用 offset。调试 ILA 时要区分 PRP1 里的地址和 AXI RAM 端口上的地址。

第五，MAX_PAYLOAD_SIZE = 128B 决定 read completion 分片上限。256-bit 数据宽度下 128B 是 4 个 beat。返回过小会增加 TLP header 开销，返回过大则可能超过 SSD/IP 当前协商的 MPS。

第六，外部 AXI4 和内部 DMA 同时访问同一个 axi_dp_ram。当前测试流程是先写 source，再发 SSD write；SSD read 完成后再读 dest 校验，避免了同一区域同时读写造成调试复杂度上升。后续真实数据流场景需要明确 buffer ownership 或做 ping-pong buffer。

第七，ssd_rw_test_ctrl 的四个 start 信号不能同拍多路触发。源码中用 start_edge_count 检查，如果一次检测到多个 start edge，会进入 ERR_MULTI_START。这让 VIO 控制更清楚：一次只启动一个阶段。

9.16 章节总结

AXI/Memory 模块把 SSD 控制器的“命令控制”和“数据搬运”拆成两条清晰路径。AXI-Lite 负责寄存器配置、命令提交和状态回读；AXI4 负责真实数据缓存；PCIe memory bridge 负责把 SSD 对 PRP 地址的 MemRd/MemWr 转换成 AXI RAM 读写。

当前实现中，axi_lite_ctl/nvme_axil_regs/cmd_source_mux 构成控制面，pcie_memory_clt_top/AXI_TLP_bridge/axi_dma/axi_dp_ram 构成数据面。16KB 测试流程通过 ssd_rw_test_ctrl 串起 AXI4 写数据、AXI-Lite 发命令、SSD 访问 memory window、AXI4 读回校验这几个步骤。对当前板级读写闭环来说，这套结构已经能够表达标准 NVMe Host memory 语义：SSD 并不直接知道 FPGA 内部 RAM，只看到一段可以按 PRP 地址访问的 host memory window。
