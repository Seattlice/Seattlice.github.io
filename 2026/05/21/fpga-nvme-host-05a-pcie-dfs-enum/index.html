5A.1 PCIe 枚举原理

PCIe 枚举本质上是 Root Complex 对下游设备配置空间的扫描和配置过程。FPGA 在这个项目里扮演的是主机侧逻辑，所以它需要主动发起 Configuration Read / Configuration Write，通过配置空间判断链路后面挂着什么设备、这个设备在哪个 BDF 上、BAR 空间需要多大，以及后续 NVMe 初始化应该访问哪个 MMIO 基地址。

一个 PCIe function 由 BDF 标识：

在 TLP 通道上，这个过程主要使用 RQ 和 RC：

配置空间里最先要看的几个 offset 是固定的：

Offset 内容 用途
0x000 Vendor ID / Device ID 判断 function 是否存在，Vendor ID = 16'hffff 表示不存在
0x004 Command Register 最后使能 Memory Space、Bus Master
0x008 Revision ID / Class Code 判断设备类型，NVMe 为 24'h010802
0x00c Header Type 判断 endpoint / bridge，以及 function 0 是否是 multi-function
0x010 BAR0 探测和配置 endpoint 的 MMIO 空间
0x018 Bridge Bus Number bridge 设备上配置 primary / secondary / subordinate bus
0x020 Bridge Memory Base / Limit bridge 设备上配置下游 memory window

普通 endpoint 的枚举比较直接：先读 Vendor ID 确认 function 存在，再读 Header Type 和 Class Code。如果 Class Code 是 NVMe，就进入 BAR 探测：向 BAR 写全 1，再读回 mask，由 mask 计算 BAR size，最后写入工程规划好的 BAR base，并写 Command Register 打开 Memory Space 和 Bus Master。

Bridge 的处理更特殊。Root Complex 不能只扫描当前 bus，因为 PCIe Switch 或桥后面的设备在 secondary bus 上。遇到 Class Code 为 24'h060400 的 PCI bridge 时，需要先给 bridge 分配 secondary bus，配置 memory window，再进入 secondary bus 扫描。深度优先枚举就是在这里体现出来的：遇到 bridge 后优先下探桥后面的 bus，子 bus 扫描结束后再回到原 bus 的下一个 device/function。

5A.2 为什么要改成深度优先枚举

早期枚举模块 pcie_us_enum_state.v 更像是面向单个固定 SSD 的初始化脚本。它默认目标 BDF 接近 01:00.0，直接读 Vendor ID、探测 BAR、写 BAR 地址、写 Command Register，然后读 NVMe CAP 验证链路。

这种方式适合第一阶段上板验证，因为最短路径清楚，能快速证明 FPGA 可以通过 RQ/RC 和 SSD 配置空间通信。但它的问题也明显：

  • 如果 SSD 不在固定 BDF 上，枚举会失败。
  • 如果链路后面挂了 PCIe Switch，旧流程不会进入 secondary bus。
  • 如果挂了多个 endpoint，旧流程无法判断哪个设备才是 NVMe。
  • 从枚举完整性看，它更像固定设备 Bring-up，不像完整的 Root Complex 枚举流程。

pcie_us_enum_dfs_state.v 的目标，是把这部分升级为更标准的硬件枚举流程：从一个起始 bus 开始扫描 BDF，遇到 PCI bridge 就配置桥并优先下探它的 secondary bus，直到找到 class code 为 NVMe 的 endpoint。

ssd_bar0_base 分配给 SSD BAR0 的基地址
ssd_bar0_size 探测出的 BAR0 空间大小
ssd_class_code SSD class code，NVMe 应为 24'h010802

参数 当前值 含义

处理策略是：

设备类型 处理方式
NVMe endpoint 记录 BDF、Vendor ID、Device ID，进入 NVMe BAR 配置流程

这里的 STRICT_ENDPOINT_FAIL 很适合调试阶段使用。如果板上预期只有 SSD，那么遇到非 NVMe endpoint 直接失败，可以快速暴露拓扑和预期不一致的问题。如果后续要支持更通用拓扑，可以把它改成 0，跳过非 NVMe 设备。

5A.6 Bridge 下探逻辑

当 class code 匹配 NVMe 后，模块进入 BAR0 探测和配置流程。

主要状态为：

流程含义如下：

  1. 先读 BAR0 原始值，判断是否是 memory BAR。
  2. 如果 BAR0 表示 64-bit BAR，则 BAR1 也参与 size 探测。
  3. 向 BAR0/BAR1 写全 1。
  4. 再读回 BAR0/BAR1，得到地址 mask。
  5. 根据 mask 计算 BAR size。
  6. 给 SSD BAR0 分配 NVME_BAR0_BASE = 0x1000_0000。
  7. 如果是 64-bit BAR，把 BAR1 上半部分写为 0。
  8. 写 Command Register 为 0x0006，使能 Memory Space 和 Bus Master。

5A.9 错误码

当前 DFS 模块定义了这些错误：

错误码 含义
ERR_NO_NVME 扫描完允许范围后没有找到 NVMe
ERR_UNSUPPORTED_EP 遇到非 NVMe endpoint，且 STRICT_ENDPOINT_FAIL=1
ERR_STACK_OVERFLOW bridge 嵌套超过 STACK_DEPTH
ERR_BUS_OVERFLOW 分配 secondary bus 超过 MAX_BUS
ERR_CFG_CPL completion status 异常
ERR_CFG_TIMEOUT 配置请求没有等到 completion
ERR_BAD_BAR BAR 类型或 size 不符合预期
ERR_BRIDGE_DISABLED 预留的 bridge 相关错误边界

这些错误码适合接入 ILA/VIO 或 AXI-Lite status register。上板时如果 enum_error=1，只看 enum_state 不够，还应该同时看 enum_error_code。

5A.11 章节总结

这部分的核心价值是把 PCIe Root Complex 枚举过程硬件化，并把 BDF 扫描、Bridge 下探、Class Code 判断和 BAR 配置放进明确的状态机流程。

可以归纳为：

这条链路的重点不是“调用 IP 自动枚举”，而是工程里明确处理了：

  • CfgRd/CfgWr 是通过 RQ 发出的。
  • Completion 是通过 RC 收回来的。
  • BDF 扫描要考虑 multi-function。
  • Bridge 后面要配置 bus number 和 memory window。
  • NVMe 是通过 class code 01/08/02 判断出来的。
  • BAR size 是通过写全 1 再读回 mask 得到的。

这说明该模块围绕 PCIe Root Complex 枚举过程展开，而不是只按固定寄存器脚本访问单个 BDF。
