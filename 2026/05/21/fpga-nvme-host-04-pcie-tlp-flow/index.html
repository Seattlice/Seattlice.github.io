4.1 四通道一句话区分

更简洁地说：

这个项目里最容易混淆的是 CQ 和 CC。CQ 不是 completion queue，CC 也不是 command completion。它们是 Xilinx PCIe IP 的 AXI-Stream 接口名称：

  • m_axis_cq / CQ：Completer Request。
  • s_axis_cc / CC：Completer Completion。

NVMe 里的 CQ 是 Completion Queue，和 PCIe IP 的 CQ 不是同一个概念。

4.2 四通道和 TLP 类型的关系

从 PCIe Transaction Layer 看，真正在线路上传输的是 TLP。RQ/RC/CQ/CC 是 Xilinx PCIe IP 在用户逻辑侧划出来的四条 AXI-Stream 通道。

当前项目中常见 TLP 类型可以这样归类：

场景 TLP 类型 出现通道 是否需要 completion 当前用途
配置空间读取 CfgRd RQ 需要 RC 枚举、读取 Vendor ID、Class Code、BAR
配置空间写入 CfgWr RQ 一般不返回数据 BAR 分配、Command Register 使能
FPGA 读 SSD BAR MemRd RQ 需要 RC 读取 NVMe CAP、CSTS 等寄存器
FPGA 写 SSD BAR MemWr RQ posted，不需要 RC 写 CC、AQA、ASQ、ACQ、doorbell
SSD 读 FPGA 队列/PRP/数据 MemRd CQ 需要 CC 读 SQE、PRP List、Write payload
SSD 写 FPGA completion/data MemWr CQ posted，不需要 CC 写 ACQ/IOCQ CQE，Read payload 写回
FPGA 响应 SSD MemRd CplD CC completion data 返回 SQE、PRP entry、memory data

4.3 RQ：Requester Request

RQ 是 FPGA 主动向 SSD 发出的 PCIe request。

在当前工程中，RQ 主要承载：

  • PCIe 枚举和 NVMe 初始化阶段的配置读写。
  • 写 SSD BAR 中的 doorbell 寄存器。
  • 队列模块发出的 ASQ/ACQ/IOSQ/IOCQ 相关 outbound request。

阶段 RQ 发出的主要请求 后续观察点
PCIe 枚举 CfgRd / CfgWr RC 返回配置 completion，枚举状态机推进
NVMe 基础初始化 MemRd / MemWr 到 SSD BAR0 CAP/CSTS 读取需要 RC，寄存器写入通常不等返回
Queue 初始化 SQ doorbell / Admin command 相关写入 SSD 后续通过 CQ 读取 ASQ
IO 运行时 IOSQ doorbell、CQ head doorbell SSD 后续通过 CQ 读取 IOSQ 或继续写 CQE

所以 RQ 本身不是“大数据通道”。对当前 16KB Write/Read 来说，RQ 主要承担通知和少量寄存器访问，真正的 16KB payload 主要走 CQ/CC 或 CQ MemWr。

4.4 RC：Requester Completion

RC 是 FPGA 主动请求之后，对端返回的 completion。

在 PCIe 枚举、配置空间读取、读取 SSD BAR 寄存器时，RC 非常重要。比如 FPGA 读 NVMe CAP、VS、CSTS 等寄存器，SSD 会通过 completion 返回数据。

4.5 CQ：Completer Request

CQ 是 SSD 主动访问 FPGA 地址空间时，PCIe IP 交给用户逻辑的 request 通道。

在当前项目中，SSD 通过 CQ 发来的请求包括：

  • 读取 Admin SQE。
  • 写入 Admin CQE。
  • 读取 IO SQE。
  • 写入 IO CQE。
  • 读取 PRP List。
  • 读取或写入 FPGA memory window。

4.6 CC：Completer Completion

CC 是 FPGA 响应 SSD 的 Memory Read 时返回的 completion。

SSD 读取 FPGA 里的 SQE、PRP List 和数据 buffer 时，都会从 CQ 发起 MemRd；FPGA 必须通过 CC 返回 CplD。

当前工程中，CC 的来源包括：

  • ASQ 返回 Admin SQE。
  • IOSQ 返回 IO SQE。
  • PRP 返回 PRP List entries。
  • Memory 返回数据 buffer。

因此，memory CplD 不应该长期只返回 32B 小包。对 16KB Write 来说，SSD 读取 FPGA data buffer 时会产生多笔 MemRd，FPGA memory 模块需要按请求长度和 MPS 限制切分 completion。每一笔 CplD 都要保持：

  • header 中 length、byte count、lower address 等字段与 payload 匹配。
  • valid && ready 时才能推进 beat 计数。
  • ready=0 时 header、data、strb、sop/eop 不能提前切到下一拍。
  • 最后一拍的 eop 和 strb 要和实际 payload 尾部一致。

这部分曾经是上板调试里比较容易暴露问题的地方。只看 data 内容不够，还要同时看 header、strb、sop/eop 和 ready/valid。

4.11 ready/valid 和流控关系

当前工程主要依赖 AXI-Stream ready/valid 背压保证 TLP 传输正确性。PCIe IP 也暴露了 credit 观测接口，例如 cfg_fc_ph/pd/nph/npd/cplh/cpld，但当前没有把它们做成完整的 credit-aware 调度。

当前实现中最重要的规则是：

状态机里的地址递增、payload beat 计数、header 切换、sop/eop 推进，都应该跟随 fire。如果 ready 被 PCIe IP 拉低，而状态机仍然继续更新 header 或 data，就会出现几类典型问题：

  • 最后一拍数据重复或丢失。
  • header 对应的是上一笔请求，data 对应的是下一笔请求。
  • eop 提前或滞后。
  • CplD length 和实际 payload beat 数不一致。
  • SSD 后续不再继续读 PRP 或 data，最终命令超时。

cfg_fc_* credit 信号可以作为后续优化方向，用来判断 PCIe IP 内部 header/data credit 是否紧张。但当前工程的基础正确性仍然优先依赖 ready/valid 和 FIFO/arbiter 的背压处理。

4.12 章节总结

本章核心是把 PCIe IP 的四条 AXI-Stream 通道和 NVMe 运行时行为对应起来：

对于当前 SSD 控制器，运行时最重要的是 CQ/CC：SSD 收到 doorbell 后，会通过 CQ 读取 FPGA 里的队列、PRP 和数据；FPGA 必须通过 CC 正确返回 CplD。NVMe Write 的 payload 是 SSD 通过 CQ MemRd 主动读走，NVMe Read 的 payload 是 SSD 通过 CQ MemWr 主动写回。
