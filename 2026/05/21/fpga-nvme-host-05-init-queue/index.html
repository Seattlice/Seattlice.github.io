5.2 PCIe 深度优先枚举与 NVMe 初始化的关系

PCIe 枚举负责找到设备和 BAR 空间，NVMe 初始化负责配置控制器寄存器。

当前工程里保留了两类枚举思路：

模块 定位
pcie_us_enum_state.v 已上板验证过的固定目标枚举路径，适合单 SSD Bring-up
pcie_us_enum_dfs_state.v 独立深度优先枚举版本，用于支持 BDF 扫描、Bridge 下探和 NVMe class code 匹配

DFS 枚举的目标是让 FPGA 更像一个轻量级 Root Complex：从 START_BUS 开始扫描 bus/device/function，通过配置读写读取 Vendor ID、Header Type 和 Class Code；遇到 PCI Bridge 时配置 secondary bus 与 memory window 并优先下探；遇到 class code 为 24'h010802 的 NVMe endpoint 后，再进行 BAR0 探测、BAR 地址分配和 Command Register 使能。

详细设计见：

对后续 SSD 控制器而言，枚举阶段的关键结果是：

  • 链路已经建立，link_up 有效。
  • 能够通过 RQ/RC 对 SSD 配置空间和 BAR 寄存器进行访问。
  • 已经知道 SSD BAR0 的访问基准，当前工程按 BAR0_BASE_ADDR = 0x1000_0000 使用。
  • nvme_init_done 表示 NVMe 控制器基础寄存器配置完成。

需要注意的是，DFS 模块当前是独立版本，pcie_us.v 主路径如果仍然例化 pcie_us_enum_state.v，则当前主路径仍以固定目标枚举为稳定版本。后续若切换成功，pcie_us_enum_dfs_state.v 输出的 ssd_bdf、ssd_bar0_base、ssd_bar0_size 可以进一步替代固定 BDF/BAR 参数。

5.3 Admin Queue 初始化流程

pcie_nvme_init_state.v 中的核心状态可以概括为：

这条链路做的事情是：

  1. 读取 NVMe CAP 等寄存器，确认控制器可以继续配置。
  2. 写 CC 寄存器关闭控制器。
  3. 轮询 CSTS.RDY，等待控制器进入非 ready 状态。
  4. 写 AQA，配置 Admin Submission Queue 和 Admin Completion Queue 深度。
  5. 写 ASQ，配置 Admin SQ 地址。
  6. 写 ACQ，配置 Admin CQ 地址。
  7. 写 CC，使能控制器。
  8. 再次轮询 CSTS.RDY，等待控制器 ready。
  9. 输出 nvme_init_done。

当前默认地址和深度为：

项目 当前值 说明
BAR0_BASE_ADDR 0x1000_0000 SSD BAR0 基地址
ADMIN_SQ_ADDRESS 0x2000_0000 Admin SQ 在 FPGA 地址空间中的窗口
ADMIN_CQ_ADDRESS 0x2000_1000 Admin CQ 在 FPGA 地址空间中的窗口
ADMIN_SQ_DEPTH 64 Admin SQ 深度
ADMIN_CQ_DEPTH 64 Admin CQ 深度

ASQ 和 ACQ 是 64-bit 地址寄存器，所以写入时使用 64-bit Memory Write，长度为 2DW：

这里要注意两个地址概念：

  • BAR0_BASE_ADDR + 0x28/0x30 是 SSD 控制器寄存器地址。
  • ADMIN_SQ_ADDRESS / ADMIN_CQ_ADDRESS 是 FPGA 暴露给 SSD 访问的队列内存地址。

也就是说，FPGA 是把“队列在主机内存中的地址”写到 SSD BAR0 的 ASQ/ACQ 寄存器里，之后 SSD 才会根据这些地址通过 CQ 通道来读写 FPGA 侧队列。

5.5 done 信号的职责

当前工程里很多 done 信号不是简单的“流程结束显示灯”，而是用于保护状态机推进。

关键 done 可以这样理解：

信号 含义
nvme_init_done NVMe 控制器寄存器配置完成，Admin Queue 可用
admin_cq_done / admin_cq_done_pulse Create IO Completion Queue 完成
nvme_queue_done / nvme_queue_done_pulse Create IO Submission Queue 完成
admin_queue_init_done 16 路 IO Queue 创建流程完成
queue_init_done 上层可认为队列初始化完成

这里的 done 大多来自 command completion 匹配，而不是状态机自己猜测命令完成。也就是说，Admin 命令真正收到 CQE 并匹配成功后，才会拉高对应 done。

电平型 done 则会在成功完成后保持：

当前 ADMIN_queue_init 使用的是 pulse 输入：

这样每一条 Create IO SQ 完成时都会产生一次 nvme_queue_done_pulse，状态机才能推进到下一路 QID。如果只看电平型 nvme_queue_done，第一条 IOSQ 完成后它会保持为 1，后面循环就可能失去“一条命令对应一次完成”的节奏。

5.6 地址一致性检查

初始化模块写给 SSD 的队列地址，必须和 CQ_route 里对 SSD 访问 FPGA 地址的解码保持一致。否则 SSD 可能已经按照 ASQ/IOSQ 地址发起 Memory Read，但 FPGA 侧 route 没有命中正确模块。

这里的方向也要分清：

SSD 行为 TLP 类型 应命中
读取 Admin SQE MemRd hit_admin_sq
写入 Admin CQE MemWr hit_admin_cq
读取 IO SQE MemRd hit_io_sq
写入 IO CQE MemWr hit_io_cq

所以初始化里写入的 ASQ/ACQ/IOSQ/IOCQ 地址，不只是软件概念，它们会直接决定后续 SSD 在 CQ 通道上访问 FPGA 时命中哪一路逻辑。

初始化阶段可以按下面顺序抓信号：

阶段 关键状态 重点信号
PCIe 枚举 pcie_enum_done debug_pcie_enum_state、RQ/RC completion
NVMe 基础初始化 nvme_init_done debug_nvme_init_state、CAP/CC/CSTS 读写
Create IOCQ admin_cq_done_pulse debug_admin_cmd_opcode=8'h05、ASQ read、ACQ write
Create IOSQ nvme_queue_done_pulse debug_admin_cmd_opcode=8'h01、QID 递增、ACQ write
队列初始化完成 queue_init_done debug_admin_queue_init_state=DONE

5.8 章节总结

这一版初始化可以作为当前工程的固定实现：

  • 已支持 Admin Queue 建立。
  • 已支持创建 1 个 IOCQ 和 16 个 IOSQ。
  • IOSQ 是硬件固定存在，后续通过 enable/disable 控制实际使用，而不是运行时真正增删硬件实例。
  • ADMIN_queue_init 当前创建的是 1 个 IOCQ，16 个 IOSQ 共享该 IOCQ。
  • 队列地址、CQ route 地址、初始化命令中的 PRP 地址必须保持一致，否则 SSD 会读写错误窗口。
