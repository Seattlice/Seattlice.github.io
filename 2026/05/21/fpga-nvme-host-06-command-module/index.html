6.1 本章定位

Command 模块是当前 NVMe Host 控制器里的“命令生命周期管理层”。它不直接解析 PCIe TLP，也不直接搬运数据 payload，而是负责把上游给出的读写/初始化请求变成 NVMe Submission Queue Entry，然后在 Completion Queue Entry 回来之后完成匹配、状态输出、资源回收和错误记录。

这个模块需要解决三个核心问题：

NVMe 允许命令乱序完成，所以不能简单地按“发出顺序”判断完成顺序。当前设计采用全局 CID 池加全局 command record 表：

这样 Admin command 和 IO command 可以共用一套命令记录与异常处理逻辑。后续从 1 个 IO queue 扩展到 16 个 IO queue 时，也不需要复制 16 份 command record，只要把 completion 汇聚到一个中心点即可。

6.2 源码文件对应关系

Command 模块不是一个单文件状态机，而是一组小模块拼起来的命令通路。当前源码对应关系如下：

command_cache.v 命令入口 FIFO 缓存上游输入的 command_type、opcode、nsid、PRP、LBA、length、CID

超时发生后，record 不会立刻释放 CID，而是把状态改为 REC_TIMEOUT：

这个策略很重要。因为 SSD 可能只是慢返回，如果 timeout 时立刻释放 CID，后面这个 CID 被新命令复用，旧 completion 再回来时就可能误伤新命令。当前实现选择“timeout 只标记，不立即释放”，等 late completion 回来后才安全释放：

含义如下：

policy 行为
0 default Admin timeout 请求 controller reset；IO timeout 先 pause queue，多次后请求 queue reset
1 report only 只记录错误，不主动提出 reset/pause
2 queue reset IO timeout 直接请求对应 queue pause/reset；Admin timeout 仍请求 controller reset

这个策略比“一条命令超时就全局重启”更细。对于 IO 队列，默认策略先把问题限制在单个 queue 上；对于 Admin queue，因为它关系到控制器管理路径，所以 timeout 会更保守地请求 controller reset。

6.14 当前实现需要注意的细节

6.14.1 CID 0 当前会被使用

command_id_pool.v 有 RESERVE_CID_ZERO 参数，但当前初始化逻辑没有使用它。因此复位后 CID 从 0 开始分配。这和前面验证阶段的设计一致。如果后续希望保留 CID 0 不用，需要补充初始化起点和 ID_TOTAL 逻辑。

6.14.2 timeout 不立即释放 CID 是正确选择

command_record.v 在 timeout 时只把状态改为 REC_TIMEOUT，不会马上释放 CID。这样可以避免旧 completion 晚回来后和新命令 CID 复用冲突。真正释放发生在 late completion 或上层明确 force release 后。

当前 command_ctl.v 里 force_release_valid 暂时接 1'b0：

也就是说，本阶段主要依靠 late completion 自然回收。如果未来要做完整 queue reset，需要由恢复模块在确认队列 drain 或 reset 完成后，再通过 force release 释放该 queue 相关 CID。

6.14.3 completion 汇聚 FIFO 深度不是无限的

command_record_round_robin 每个端口 FIFO 深度为 16：

如果某个 CQ 短时间内 completion 太密集，而 command_record 处理不过来，就可能出现 cpl_fifo_overflow。当前这个异常会进入 ERR_FIFO_OVERFLOW，并被 command_error_ctl 视为高风险错误。

6.14.4 Admin path 里的 i_lba 是复用字段

在 command_construct.v 里，Admin Create IO CQ/SQ 使用 i_lba 填 DW10/DW11。这是为了复用统一 command input 接口，不代表 Admin 命令真的有 LBA。看波形时如果看到 Admin 命令的 i_lba 有值，应该按 CDW10/CDW11 解释。

6.14.5 io_id_valid 是兼容旧接口

command_ctl.v 既有新的 16 路 IO completion bus，也保留了旧的单路 io_id_valid/io_id_data/io_cpl_queue_id 输入。旧接口会覆盖 port 1，所以完整 16 queue 集成时应统一使用 bus 接口，旧单路输入保持无效。

6.14.6 issue_context 当前暂时为 0

command_record 支持 CONTEXT_WIDTH=64，但当前 command_ctl.v 例化时：

也就是说 record 表已经预留 context 字段，但当前还没有把 PRP index、AXI command descriptor、请求来源等信息写进去。后续如果要增强 debug，可以把 cmd_source、queue_hint、PRP window id 或 AXI-Lite command seq 放进 context。

6.15 章节总结

Command 模块当前已经完成了 NVMe 命令生命周期的主要硬件化：

  • 用 command_cache 缓存上游命令，吸收后级 backpressure。
  • 用 command_id_pool 维护全局 CID 池，CID 当前从 0 开始分配。
  • 用 command_construct 生成 512-bit SQE，并按 Admin/IO 分流。
  • 在 SQE 真正被队列 ready/valid 接收时写入 command_record。
  • 用 command_record_round_robin 汇聚 1 Admin + 16 IO completion。
  • 用 command_record 按 CID 处理乱序 completion、queue mismatch、lookup miss、late completion 和 timeout。
  • 用 command_error_ctl 统计错误并产生 pause/reset 请求。
  • 通过 AXI-Lite 暴露 done、last error、timeout count、mismatch count 和 watchdog 配置。

从系统角度看，Command 模块处在 AXI-Lite/初始化命令源和 Queue 模块之间。它向下游提交 SQE，向上游返回命令完成状态，并把 NVMe 的乱序 completion 变成可追踪、可回收、可调试的硬件状态。
