FPGA NVMe Host 项目记录（二）：PCIe、NVMe 与 IP 配置

这一篇主要记录项目里用到的 PCIe / NVMe 基础概念、Xilinx PCIe IP 配置，以及我在设计时反复确认的一些注意事项。它不是完整协议手册，更像是围绕当前工程整理出来的工作笔记。

PCIe 和 NVMe 的关系

NVMe SSD 底层通过 PCIe 传输数据，但 NVMe 和 PCIe 不是同一层协议。

在普通 PC 里，CPU 和操作系统驱动完成这些工作：

  • PCIe 枚举，找到 SSD 设备和 BAR 空间。
  • 配置 NVMe 控制器寄存器。
  • 建立 Admin Queue 和 IO Queue。
  • 把 NVMe command 写入 Submission Queue。
  • 写 doorbell 通知 SSD 取命令。
  • 解析 Completion Queue Entry，回收命令资源。

本项目把这些流程下沉到 FPGA 内部，用 RTL 完成主机侧控制逻辑。

FPGA 在系统中的角色

从协议角色看，FPGA 在本项目中承担的是轻量级 Host Controller 的角色。

这也是后续博客中应该坚持的项目边界：轻量级 NVMe Host Controller，而不是完整商用 NVMe 协议栈。

PCIe TLP 四条通路

Xilinx PCIe IP 对用户逻辑暴露四条 AXI-Stream 通道：

通道 全称 从 FPGA 视角的方向 本项目用途
RQ Requester Request FPGA 主动发请求 枚举、初始化、doorbell、outbound write/read
RC Requester Completion 对端返回 FPGA 请求的 completion 配置空间和寄存器读取返回
CQ Completer Request SSD 主动访问 FPGA 地址空间 SSD 读 SQE、读 PRP、读数据、写 CQE
CC Completer Completion FPGA 响应 SSD 的 non-posted request 返回 SQE、PRP list、memory read data

这里需要先明确：运行时的 NVMe IO 并不是 FPGA 一直主动把数据推给 SSD，而是 SSD 根据队列和 PRP 地址主动访问 FPGA 暴露出来的地址窗口。

NVMe Queue、Doorbell、CID 和 PRP

最小 NVMe 系统需要四类队列：

队列 作用
Admin Submission Queue, ASQ 提交管理命令，例如 Identify、Create IO Queue
Admin Completion Queue, ACQ 接收管理命令完成信息
IO Submission Queue, IOSQ 提交读写命令
IO Completion Queue, IOCQ 接收读写命令完成信息

当前工程最终按下面方向组织：

NVMe 队列本身位于主机内存地址空间。SSD 不会自动知道 FPGA 已经写入了新的 SQE，所以主机写完 SQE 后需要写 doorbell。

CID 用来把发出去的命令和回来的 completion 对应起来。SSD 可能乱序完成不同命令，所以不能只按发送顺序判断哪条命令完成。当前 command_ctl 中围绕 CID 做了分配、记录、匹配、回收和错误统计。

PRP 则是 NVMe 的物理页地址描述方式：

当前 16KB 测试对应 4 个 4KB page。prp_ctl 不预先把完整 PRP list 写入外部 DDR，而是在 FPGA 内部维护 CID 到 page 信息的上下文。当 SSD 发起 PRP list read 时，由 prp_ctl 根据地址和 CID 返回对应 PRP entry。

当前地址窗口

当前默认规划可以概括为：

地址窗口 默认地址 用途
BAR0 0x1000_0000 SSD 控制寄存器和 doorbell 基地址
ASQ 0x2000_0000 Admin Submission Queue
ACQ 0x2000_1000 Admin Completion Queue
IOSQ 0x2000_2000 IO Submission Queue 区域
IOCQ 0x2001_2000 IO Completion Queue 区域
Memory Window 0x3000_2000 FPGA 数据缓存窗口，当前 size 为 64KB
PRP List Window 0x9000_0000 虚拟 PRP list 地址窗口

这些地址需要在初始化、CQ route、PRP、memory controller 中保持一致。如果某个模块里的地址常量没有同步更新，SSD 发出的 MemRd/MemWr 可能会打到错误窗口，表现为命令卡住、CQE 不回来或数据校验失败。

Xilinx PCIe IP 手册学习

当前使用的是 Xilinx UltraScale PCIe Integrated Block，主要参考 PG156。这个 IP 的用户侧接口可以分成三类：时钟和复位接口、Configuration / Management 相关接口、PCIe Transaction 相关 AXI-Stream 接口。

时钟侧需要外部 PCIe 参考时钟，当前工程按 100 MHz 参考时钟处理；用户逻辑侧使用 IP 输出的 user_clk，当前配置为 125 MHz。

CFG 相关接口用于读取或控制 PCIe IP 自身状态，例如配置空间、协商出的 MPS/MRRS、链路状态、流控信息等。它不是 SSD 数据读写的主路径。

真正和 TLP 数据收发有关的是 RQ、RC、CQ、CC 四条 AXI-Stream 通路。

KU060 板卡上有多个 PCIe hard block 位置，当前工程使用的是与板卡走线匹配的位置。之前上板调试时，PCIe block location 选择不一致会导致 link 训练失败，所以这部分要和板卡约束一起确认。

NVMe 初始化相关记录

当前学习和实现主要参考 NVMe 1.3d。对当前项目来说，关键不是版本号本身，而是理解下面几类内容：

  • Controller Register：CAP、VS、CC、CSTS、AQA、ASQ、ACQ、doorbell。
  • Admin Command：Identify、Create IO CQ、Create IO SQ。
  • IO Command：Read、Write。
  • Queue Entry：SQE / CQE 格式。
  • PRP：数据页地址描述。

队列空满判断不能只看地址是否相等，还要结合 head、tail、phase bit 等上下文。

NVMe PRP 地址和 page 关系很紧密。当前项目里以 4KB page 为基本粒度组织数据窗口和 PRP list。

NVMe SQE 是 64B，里面包含 opcode、CID、NSID、PRP1、PRP2、SLBA、NLB 等字段。当前工程的 command_construct 就是把上游读写参数拼成对应 SQE。

初始化阶段最基础的一组寄存器访问如下。实际工程中的地址和队列深度以当前地址规划为准。

步骤 寄存器 偏移地址 写入值示例 说明
1 CC 0x14 0x00000000 关闭控制器，保证 AQA/ASQ/ACQ 可重新配置
2 AQA 0x24 由队列深度决定 Admin Queue 深度字段是 0-based
3 ASQ_L 0x28 ASQ 低 32 位地址 指向 FPGA 暴露给 SSD 的 ASQ 地址

5 ACQ_L 0x30 ACQ 低 32 位地址 指向 FPGA 暴露给 SSD 的 ACQ 地址

7 CC 0x14 由 MPS、IOSQES、IOCQES 等字段决定 使能 NVMe 控制器
8 CSTS 0x1C 只读 轮询 CSTS.RDY，等待控制器 ready

Xilinx IP 配置要点

当前 PCIe IP 的一些端口和侧边带接口可以按下面方式理解。

时钟、复位和链路状态

板载 PCIe 100 MHz 差分参考时钟进入 IBUFDS_GTE4，PCIe IP 输出的 user_clk 作为用户逻辑主时钟，当前为 125 MHz。用户逻辑复位跟随 user_reset，枚举、初始化和 IO 测试都应等待 user_lnk_up 为 1 后再启动。

当前项目不通过 cfg_msg_transmit_* 主动发送 PCIe Message，也不处理 cfg_msg_received_* sideband message。当前只使用一个 PCIe Root Port IP，不共享底层 GT/PLL 资源，因此 shared logic 相关输出也不接入用户逻辑。

设计注意事项

这部分是当前工程里最容易在上板时暴露问题的设计点。

时钟和复位边界

sys_clk / sys_clk_gt 来自板卡 PCIe 参考时钟路径；user_clk 是 PCIe IP 输出给用户逻辑的 AXI-Stream 时钟。RQ/RC/CQ/CC、NVMe 初始化、command、queue、PRP、memory 等主逻辑默认都放在 user_clk 域。用户逻辑复位跟随 user_reset，内部可转换成 user_rst_n。

Root Port 和 Endpoint 角色

当前 FPGA 是 Root Port，SSD 是 Endpoint。Xilinx IP 配置页里的 PF0 BAR 是 FPGA 这个 PCIe 端口自己的 BAR；NVMe 初始化里真正要访问的是 SSD 配置空间里的 BAR0。SSD BAR0 分配完成后，FPGA 才能通过 SSD_BAR0_BASE + offset 访问 CAP、CC、CSTS、AQA、ASQ、ACQ、doorbell 等寄存器。

MPS / MRRS 和数据分包

PCIe IP 配置页里填写的是能力上限，实际运行时还要看链路协商和配置后的 MPS/MRRS。上板时如果实际 MPS 是 128B，completion data 和 Memory Write payload 就要按这个边界切分。

当前用户侧 AXI-Stream 是 256-bit，一拍 32B。为了效率，数据 payload 不应长期只发 32B 小包；但每个 TLP 也不能超过实际 MPS。更合理的做法是按 MPS 聚合，例如 128B 对应 4 拍 256-bit 数据。

AXI-Stream 计数跟随 fire

RQ、RC、CQ、CC 都是 ready/valid 协议。内部状态机推进、地址递增、payload beat 计数、last 生成，都应跟随：

上板时 ready 被 PCIe IP 拉低是正常现象，如果状态机在 ready=0 时继续修改 header/data，很容易出现最后一拍错位、length 不一致或 completion 内容重复。

Queue、Doorbell 和 PRP

如果做 doorbell 聚合，需要保证聚合只影响通知频率，不改变队列里 SQE 已经可读这个事实。CQ 侧还要注意 phase bit，head wrap 时 phase 翻转需要用寄存器保存，不能只靠单拍组合判断。

当前 PRP 设计适合连续地址和固定 memory window 的读写测试。后续如果要支持随机物理页、复杂 scatter-gather 或不同 namespace 的大规模并发，需要把 PRP list 做成真正可缓存、可索引、可回收的 RAM 表。

当前边界

当前版本更适合作为轻量级 NVMe Host Controller 原型：

  • 支持固定地址窗口下的 Admin Queue、IO Queue 和 16KB 读写闭环。
  • 支持基础多队列结构、CID 匹配、PRP list 响应和状态统计。
  • 不把 MSI-X、完整 PRP RAM、完整错误恢复、复杂电源管理作为当前主线。

这个边界在写博客和准备面试时也要讲清楚。项目的亮点是从 PCIe TLP 到 NVMe 队列的数据闭环，而不是声称已经实现完整商用 SSD 驱动。
