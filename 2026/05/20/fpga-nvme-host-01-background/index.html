1.1 项目目标

这个项目的目标是在 FPGA 内部实现一套轻量级 NVMe over PCIe Host Controller。它不是简单调用 PCIe IP，也不是只做一条 DMA 搬运通路，而是在 Xilinx PCIe Gen3 IP 提供的 AXI-Stream TLP 接口之上，用 RTL 完成 SSD 访问所需的主机侧控制逻辑。

我这里主要做的事情是：把 Xilinx PCIe Gen3 IP 输出的 AXI-Stream 数据流解析成 TLP header 和 payload，再按照地址、TLP 类型和方向路由到 NVMe 初始化、队列、PRP、数据缓存等模块。不同模块处理完成后，再通过 RQ/CC 等通路返回对应的 TLP，或者更新 FPGA 内部的 memory window，从而完成 FPGA 与 SSD 之间的数据交互。

1.2 项目背景

在普通 PC 系统中，NVMe SSD 由 CPU、操作系统和驱动共同管理。PCIe 枚举、NVMe 初始化、队列建立、PRP 地址管理、命令提交和 completion 回收都由软件栈完成。

对于数据采集、实时缓存和硬件加速这类场景，如果每次 IO 都依赖 CPU 参与，会带来软件调度开销，也会增加系统时延的不确定性。这个项目尝试把 SSD 控制链路的一部分下沉到 FPGA 内部，让 FPGA 作为 PCIe Host/Root 侧逻辑直接控制 NVMe SSD。

项目里 FPGA 负责：

  • 建立 PCIe 链路，并完成 SSD 枚举。
  • 配置 NVMe 控制器寄存器。
  • 建立 Admin Queue 和 IO Queue。
  • 维护 command、queue、PRP 和 memory window。
  • 通过 AXI-Lite 暴露控制寄存器，通过 AXI4 暴露数据缓存窗口。

这样做的直接意义是：外部采集或计算模块可以先把数据写入 FPGA 侧 AXI4 memory window，再由 FPGA 触发 NVMe Write 写入 SSD；读数据时也可以由 FPGA 触发 NVMe Read，把 SSD 数据读回 AXI4 memory window。对于图像数据、雷达原始数据等连续大带宽数据流，这种结构可以减少 CPU 参与，为后续长时间记录和回放打基础。

当前工程中几个调试接口的定位如下：

  • AXI-Lite：配置寄存器、选择命令来源、触发读写命令、读取状态和错误信息。
  • AXI4：访问 FPGA 侧数据缓存窗口，测试时由 ssd_rw_test_ctrl 写入或读出 16KB 数据。
  • VIO：板级调试阶段手动控制测试开始，例如 rw_write_start、rw_write_start_command、rw_read_start_command、rw_read_start。
  • ILA：观察 PCIe TLP、PRP、CQ route、completion、done pulse 等关键路径。

1.4 当前实现范围

当前已经实现和验证过的内容包括：

  • PCIe 链路与 NVMe 初始化流程。
  • Admin Queue 和 IO Queue 建立。
  • NVMe Read/Write 命令下发。
  • PRP1/PRP2 和 PRP List 生成。
  • SSD 对 SQE、PRP List、data buffer 的 PCIe 访问响应。
  • Completion 回收和 CID 匹配。
  • AXI-Lite 控制寄存器和 AXI4 数据缓存接口。

当前还没有把所有企业级 NVMe 特性都做完整，后续可以继续扩展：

  • 更完整的 OS 级物理页管理。
  • MSI-X 中断接收和中断路由。
  • 更细粒度的 PCIe credit-aware 调度。
  • AER、Abort、SGL、metadata、namespace 管理等高级 NVMe 特性。

这篇项目总结后续会围绕“能上板跑通的轻量级 NVMe Host Controller”来展开，不把它描述成完整商用 NVMe 协议栈。
