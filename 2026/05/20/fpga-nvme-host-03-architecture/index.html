3.1 架构概览

这一章主要说明当前 SSD 控制器在工程中的位置，以及各个模块之间的数据流关系。前两章已经说明了 PCIe/NVMe 的基本概念，这里开始从实际工程结构展开。

当前工程可以理解成三层：

系统详细架构框图如下：

这一张图里需要先看清楚的关系是：FPGA 不是把 SSD 当成一个普通 FIFO 去写，而是作为 PCIe Root Port 建立 NVMe Host 逻辑。命令、队列、PRP 和数据缓存都位于 FPGA 侧，SSD 在执行命令时会反过来通过 PCIe 访问 FPGA 暴露出来的地址窗口。

3.2 顶层数据流

从顶层看，系统里有六个主要对象：

这里有几个职责关系需要先说明清楚：

  • pcie_us 负责 Xilinx PCIe IP、Alex Forencich PCIe 适配层、枚举和基础 NVMe 初始化路径。
  • SSD_ctl 负责运行时 NVMe command、queue、PRP、memory window 和 TLP route。
  • nvme_command_ctl 主要提供初始化阶段的 Admin 命令来源。
  • axi_lite_ctl 提供运行时软件/测试侧命令入口和状态寄存器。
  • ssd_rw_test_ctrl 是板级测试状态机，用 VIO 触发 16KB 写读闭环。
  • external AXI memory window 是 FPGA 侧可被 SSD 访问的数据缓存窗口。

当前 SSD_ctl 运行时主数据路径主要围绕 RQ/CQ/CC 展开。RC 仍然是 PCIe requester completion 方向，但枚举和基础初始化相关的 completion 主要在 pcie_us 内部路径处理，SSD_ctl 本身不消费运行时 RC completion。

3.3 总体层级

当前工程主层级可以按下面理解：

top_v 是上板顶层，主要做三件事：

  • 接入 PCIe 物理引脚和参考时钟。
  • 例化 ssd_top，得到完整 SSD 控制器。
  • 例化 ssd_rw_test_ctrl 和 VIO，用于进行板级测试。

ssd_top 是 SSD 控制器封装层，负责把 PCIe IP、初始化逻辑、主控制器和 AXI/AXIL 接口接起来。

SSD_ctl 是 NVMe host control 的核心顶层，负责命令、PRP、队列、memory window 和 TLP route。

3.5 PCIe TLP 方向与系统职责

四条 PCIe AXI-Stream 通路在当前系统中的方向如下：

从工程角度看：

通道 当前职责
RQ FPGA 主动发请求，主要用于枚举、寄存器访问、doorbell 写入
RC 返回 FPGA 主动请求的 completion，当前主要在 pcie_us 内部用于枚举和初始化
CQ SSD 主动访问 FPGA 地址空间，包括读 SQE、读 PRP、读数据、写 CQE、写读回数据
CC FPGA 响应 SSD 的 Memory Read，返回 SQE、PRP entry 或 data buffer 内容

RC 这条路径需要区分清楚：不是系统不用 RC，而是当前 SSD_ctl 运行时主数据路径不直接消费 RC completion。PCIe 枚举和 NVMe 基础初始化仍然需要 RQ/RC 完成配置读写和寄存器读取。

3.6 从工程视角划分数据面

当前系统可以分成六个面。

3.6.1 PCIe TLP 面

对应模块：

作用：

  • 把 Xilinx PCIe IP 的 AXI-Stream RQ/RC/CQ/CC 转成工程内部 TLP 信号。
  • 对 FPGA 主动发出的请求做 RQ 仲裁。
  • 对 SSD 发来的 MemRd/MemWr 做 CQ 地址路由。
  • 对 queue/memory/PRP 返回的 CplD 做 CC 仲裁。

3.6.2 控制面

对应模块：

作用：

  • AXI-Lite 寄存器写入命令参数。
  • 选择 legacy init command、AXI-Lite command 或外部 command descriptor。
  • 读取 done/error/status/counter。
  • 在板级调试时由 VIO 触发 16KB 测试。

3.6.3 初始化面

对应模块：

作用：

  • pcie_us 内部负责 PCIe 枚举和 NVMe 基础初始化，并输出 init_done/init_error。
  • nvme_command_ctl 在初始化完成后发出 Admin 队列创建相关命令。
  • 上层通过 queue_init_done 判断 Admin Queue 和 IO Queue 初始化是否完成。

3.6.4 命令面

对应模块：

作用：

  • prp_ctl 先拿到 command 参数，申请 CID，计算 PRP1/PRP2。
  • command_ctl 生成 512-bit NVMe SQE。
  • command_record 记录 CID 对应的 queue、opcode、状态和上下文。
  • CQE 回来后通过 CID 匹配完成，并释放资源。

3.6.5 队列面

对应模块：

作用：

  • 管理 ASQ/ACQ 和 IOSQ/IOCQ。
  • IO Queue 数量参数为 IO_QUEUE_COUNT = 16。
  • Admin SQ/CQ 深度为 64，IO SQ 深度为 64，IO CQ 总深度为 1024。
  • 支持 IO queue enable、dispatch mode、fixed queue id、queue weight。
  • 负责 SQ doorbell 和 CQ head doorbell。

3.6.6 PRP 与数据缓存面

对应模块：

作用：

  • PRP 模块生成 PRP1/PRP2，并在 SSD 读取 PRP List 时返回对应 page 地址。
  • Memory 模块响应 SSD 对 FPGA memory window 的 MemRd/MemWr。
  • 外部 AXI4 可以直接写入或读取这片 memory window。

3.7 FPGA 与 SSD 的运行时交互流程

从 NVMe 的角度看，一条命令不是 FPGA 单向把数据推给 SSD，而是 FPGA 提交命令后，SSD 再根据队列和 PRP 地址主动访问 FPGA。

3.7.1 命令提交

FPGA 构造 64B NVMe SQE，并写入 ASQ 或 IOSQ。随后 FPGA 通过 RQ 发送 Memory Write TLP，写 SSD BAR0 中对应队列的 SQ Tail Doorbell。

这一阶段的关键是：doorbell 不能早于 SQE 可读。否则 SSD 看到 doorbell 后来读 SQE，可能读到旧内容或无效内容。

3.7.2 SSD 获取命令

SSD 收到 doorbell 后，会通过 CQ 发起 Memory Read，请求读取 FPGA 地址空间中的 SQE。

FPGA 的 CQ_route 根据地址把请求路由到 queue 模块，queue 模块取出 SQE 后通过 CC_round_robin 返回 CplD。到这里 SSD 才真正拿到 NVMe 命令。

如果是 NVMe Read，数据方向是 SSD 到 FPGA：

因此写命令和读命令在 PCIe 上的“主动方”并不完全一样。写 SSD 时，实际 payload 是 SSD 主动读走；读 SSD 时，payload 是 SSD 主动写回。

3.7.4 命令完成

数据传输完成后，SSD 会构造 CQE，并通过 CQ Memory Write 写入 ACQ 或 IOCQ。队列模块解析 CQE 后，将 CID、SQID、status 送到 command completion 路径。

command_record 用 CID 找到原来的命令上下文，判断 opcode、queue_id、status 是否匹配。匹配成功后释放 CID，并产生 io_wr_done、io_rd_done 或 Admin done pulse。

处理完 CQE 后，FPGA 还需要更新 CQ Head Doorbell，告诉 SSD completion queue 中的 entry 已经被消费。

3.8 当前关键地址窗口

最新工程中的默认地址规划如下。

区域 地址 用途
SSD BAR0 base 0x1000_0000 FPGA 访问 SSD BAR 和 doorbell 的基地址
Admin SQ 0x2000_0000 SSD 读取 Admin SQE
Admin CQ 0x2000_1000 SSD 写入 Admin CQE
IO SQ 0x2000_2000 SSD 读取 IO SQE
IO CQ 0x2001_2000 SSD 写入 IO CQE
Data memory base 0x3000_2000 FPGA/SSD 共享数据窗口起点
Write source buffer 0x3000_2000 16KB Write 测试源 buffer
Read destination buffer 0x3000_8000 16KB Read 测试目标 buffer
Memory window size 64KB 0x0001_0000
PRP list base 0x9000_0000 SSD 读取虚拟 PRP List 的地址窗口
PRP list window size 32MB 0x0200_0000

这里需要区分几个概念：

  • SSD BAR0 base 是 FPGA 访问 SSD 控制寄存器和 doorbell 的基地址。
  • ASQ/ACQ/IOSQ/IOCQ 是 FPGA 暴露给 SSD 的队列内存地址。
  • Data memory base 是 SSD 根据 PRP 访问的实际数据窗口。
  • PRP list base 是一个虚拟 PRP list 窗口，SSD 读这个窗口时由 prp_ctl 动态返回 PRP entry。

这些地址必须在 NVMe 初始化、CQ route、queue、PRP、memory controller 中保持一致。只改一个模块会造成 SSD 请求打到错误窗口，表现为命令卡住、CQE 不回来或数据校验失败。

3.11 章节总结

当前架构已经预留或实现了几个扩展点：

  • IO Queue 可扩展到 16 路，队列调度支持 fixed、RR、WRR 等策略。
  • command 模块有 CID 池和 command_record，可以处理乱序 completion。
  • queue 模块加入 doorbell 聚合思路，减少频繁 doorbell Memory Write。
  • PRP 模块支持虚拟 PRP list window，当前适合连续地址和固定窗口测试。
  • AXI-Lite 暴露配置和状态寄存器，后续可以接入更多错误统计和调试状态。

本章可以归纳为：

  • 运行时主数据路径主要是 RQ/CQ/CC，SSD_ctl 不直接消费 RC completion。
  • PRP list 还不是完整 OS 级物理页表管理。
  • MSI-X 接收、完整错误恢复、AER/Abort 等高级功能不是当前主线。
  • 当前板级测试以 16KB 写读闭环和固定 memory window 为主。

这个版本更适合作为轻量级 NVMe Host Controller 原型，重点展示从 PCIe TLP 到 NVMe 队列、PRP、数据窗口和 completion 回收的完整硬件闭环。
