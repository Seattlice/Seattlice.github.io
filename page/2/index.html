CQ调试

抓到了CQ的端口数据：

cq_data:0000000000000000000000000000000000070000000000100000000020000000

cq_tuser:1ffff7dee00100000000ff

cq_tkeep:0f

继续深入抓取数据：

  • **cpl_fmt**** = 1 (**3'b001**)**

    • 含义：3DW Header, No Data。
    • 这表示这是一个不带数据的请求（也就是读请求），且使用 32 位地址。

  • **cpl_type**** = 0 (**5'b00000**)**

    • 含义：Memory Read / Write。
    • 结合 fmt=1 (No Data)，这确定无疑是一个 Memory Read (内存读请求)。

  • **cpl_lowaddress**** = **20000000** (Hex)**

    • 含义：读取的目标地址。
    • 分析：这正是你之前配置的 Admin SQ Base Address (**32'h2000_0000**)。
    • 结论：SSD 正在准确地访问你分配给它的队列内存。

  • **cpl_length**** = **010** (Hex)**

    • 含义：请求读取的数据长度（单位是 Dword / 4字节）。

    • 换算：Hex 010 = Decimal 16。

    • 字节数：16×4 Bytes=64 Bytes。
      16×4 Bytes=64 Bytes16 \times 4 \text{ Bytes} = \mathbf{64 \text{ Bytes}}

    • 结论：64 字节正是 NVMe 一条命令的标准长度！

现在CC端口发骚那个成功了。得到CQ端口回应的CQE数据如下：

ACQ_rx_req_tlp_hdr_reg[127:0]：

60000004000000ff0000000020001000

ACQ_rx_req_tlp_data_reg[255:0]：0007000000000010000000002000000040170000000000010000000000000000

解析如下：

  • **DW0: ****60000004**

    • Fmt = 011 (4DW 头，带数据)
    • Type = 00000 (Memory Write)
    • Length = 0x004 (4 个 DWord = 16 字节)
    • 含义：SSD 发起了一个 DMA 内存写操作，写入长度恰好是 16 字节（标准的 1 个 NVMe CQE 大小）。

  • **DW1: ****000000ff**

    • Requester ID = 0x0000 (SSD 控制器本身的 ID)
    • First BE / Last BE = f / f (4个DWord全部有效)

  • **DW2: ****00000000** (64位物理地址的高 32 位)

  • **DW3: ****20001000** (64位物理地址的低 32 位)

    • 含义：SSD 精准地将回信写到了你指定的 **ADMIN_CQ_ADDRESS** (**0x20001000**) 处！
  1. 解析 TLP Data (NVMe 完成包 CQE)

**ACQ_rx_req_tlp_data_reg = [垃圾数据] 40170000 00000001 00000000 00000000**

因为 Length 是 4 DW，所以 256-bit 总线上只有低 128-bit (最右边的 4 个 DWord) 是有效的 CQE 数据。我们按小端序（右边是 DW0）拆解：

  • **DW0: ****00000000** (Command Specific，通常为 0)
  • **DW1: ****00000000** (Reserved)
  • **DW2: ****00000001** (SQ Head Pointer & SQ Identifier)
    • SQ Identifier = 0 (这是 Admin SQ)
    • SQ Head Pointer [15:0] = 1
    • 含义：SSD 告诉你：“Admin 队列里索引为 0 的那个命令，我已经取走并执行完了，现在我的 Head 指针移动到 1 了。”

最关键的 DW3: 40170000 (状态与结果)

这个词包含了命令的执行结果，我们将 4017（高16位）转为二进制：
0100_0000_0001_0111

根据 NVMe 协议对应位段：

  • **CID**** (Command ID) [15:0]**: 0000 (回应你 CID=0 的那条命令)
  • **P**** (Phase Tag) [16]: 1 (这是最低位的 1。代表这个 CQE 是最新有效**的！)
  • **SC**** (Status Code)**: 00001011 = 11 = **0x0B**
  • **SCT**** (Status Code Type)**: 000 (Generic Command Status)
  • **M**** (More Bit) [30]**: 1 (表示有更多的错误日志可以读取)

结论：SSD 拒绝了你的命令，报错代码是 Generic Status **0x0B**。

修改了这个问题之后还是IOSQ没有收到东西，继续查询觉得可能是数据有问题：

强制绑定了地址，而且不管3DW，还是4DW，低位才是最后的地址才对，所以这里出错了，必须得到修正。

而且经过仿真与查询，TLP的BYTECOUNT是十一位宽的数据，但是如果一次性传4096就会造成错误，因此必须在内部赋值为12位宽，然后外侧改为11位宽才是正确的做法，这样做可以得到正确的仿真结果，每一次传输128字节数据

对于这个模块，还需要优化的地方如下：

  1. 所有的读取请求地址，必须是 32 字节（256-bit）对齐的（即地址的最低 5 位永远是 00000）。
  2. 所有的单次读取请求长度，不要跨越 4KB 边界。
